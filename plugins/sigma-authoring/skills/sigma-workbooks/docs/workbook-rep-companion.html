The workbook rep — an agent edits a Sigma workbook as files

No middleware and no new interface: the spec becomes a directory of small files, the agent works with the file tools it is already best at, and a render loop means it sees what it built before calling it done.

STRUCTURE

The element is the unit of work

Sigma's spec API moves whole workbooks — every small edit drags the full document through an agent's context, and parallel work is a merge hazard. The rep explodes the spec into one file per element and reassembles it on push.

Nothing is a new format: every file is a verbatim slice of the spec, so existing spec documentation applies unchanged. A 20-element workbook becomes twenty ~600-byte files; an edit touches exactly one. Add an element = add a file. Delete = delete it. Reorder = rename the prefix.

LAYOUT

A workbook, as a directory

<rep>/ workbook.yaml name, folderId, … pages/ 010-overview/ prefix = order _page.yaml page identity _layout.xml layout XML block 010-revenue-kpi.yaml one element each 020-sales-by-region.yaml renders/ agent-readable PNGs .sigma/ snapshot + manifest

The element-level abstraction lives in the file layout, not in a process — the agent's native Read / Edit / Glob / grep are the API.

THE LOOP
pull
edit files
status
push
verify
render & look

push reassembles the files and writes through four gates:

  1. Drift check — if the server spec changed since pull (a UI edit), push aborts and shows exactly what would be overwritten. No silent clobbering, ever.
  2. Diff preview — the element-level change set about to apply.
  3. Validation — static formula/shape checks before any network write.
  4. Readback — re-GET after the PUT, refresh the snapshot, report anything the server normalized.
$ wb-rep status my-dashboard/ page "Pulse": ~ element "Clones per day" [columns, yAxis, color] + element "Star events per month" $ wb-rep push my-dashboard/ # drift-check · diff · validate · readback rendered "Pulse" -> renders/pulse.png

The API still receives a standard full-spec PUT — element semantics arrive on top of the existing endpoint, today. Reassembly is byte-exact: an untouched rep pushes back as a no-op.

FEEDBACK

The render loop — the agent looks at its work

A spec that POSTs cleanly and compiles cleanly can still be wrong. render exports any page or element as a PNG; the agent reads the image and critiques it against the plan. Acceptance is "the pixels look right" — not "the API returned 200."

Failure classes only this gate has caught:

  • a style the API accepted and silently dropped — element rendered with no background
  • a formula error inside {{…}} text templating — invisible to both POST and SQL compile
  • a chart numerically correct and visually useless
  • KPI panels clipping at real viewport widths

BEYOND THE SCHEMA

The verified-behavior catalog

The OpenAPI says what shapes parse; it can't say what the platform does. The rep's companion reference keeps a catalog where every entry is live-verified against the API and marked ✓/✗ with the exact error and the workaround — which idioms are UI-only, which fields render although undocumented, which writes are accepted-then-dropped. When a render exposes a new behavior, it lands in the catalog the same day, and every future build inherits it.

WHAT THE STRUCTURE BUYS
Parallel builds

Element files are independent — one sub-agent per element, zero merge conflicts, one atomic push.

Create mode

A rep authored from scratch POSTs a new workbook through the same gates. The directory is the design doc.

Git-native

PRs show changed: revenue-kpi.yaml, not a 2,000-line spec diff. Workbooks-as-code for free.

Zero install

One dependency-free script (Ruby stdlib) plus the agent's own file tools. No server, no daemon.
